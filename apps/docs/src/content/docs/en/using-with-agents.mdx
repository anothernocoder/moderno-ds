Moderno ships a surface built for coding agents, not just for humans reading
docs: a local MCP server that answers with the API actually installed in your
project, plus a deterministic validator that catches mistakes before an agent
calls the job done. This page covers setup, the end-to-end loop, and the rules
an agent should follow while it's in a Moderno codebase.

## Configure the MCP server

`@moderno/mcp` is a local stdio server — no account, no hosted endpoint. Run it
with your package manager's dlx runner:

<Install pkg="@moderno/mcp" mode="cli" />

Most agent tools (Claude Code, Cursor, and others that speak MCP) read server
definitions from a JSON config. Add an entry for `moderno`:

```json
{
  "mcpServers": {
    "moderno": {
      "command": "npx",
      "args": ["-y", "@moderno/mcp"]
    }
  }
}
```

The server discovers `moderno.agent.json` from `node_modules/@moderno/*` **once,
at startup** — so it must run from (or be pointed at) the project that has
Moderno installed. Because it reads what's actually installed, a project pinned
to an older `@moderno/react` gets that version's API back, not whatever shipped
most recently.

## The agent loop

The tools are read + verify, not write — an agent still writes the code itself
and shells out to `@moderno/cli` for scaffolding. The MCP's job is to ground
that code in the real API and then check it.

### 1. `search_components` — find the primitive by intent

Don't guess a component name; describe the UI need.

```
search_components({ query: "modal blocking decision", framework: "react" })
→ matches: [{ name: "Dialog", score: 0.82, … }, { name: "Button", score: 0.11, … }]
```

### 2. `get_component_api` — get its real props

Read the props, `data-part`s, and variants from the **installed** version
before writing any markup.

```
get_component_api({ name: "Dialog", framework: "react" })
→ { package: "@moderno/react", version: "1.4.0", component: { props: […], parts: […] } }
```

### 3. `get_examples` — get a working snippet in your framework

Never hand-translate a React example into Vue or Svelte syntax — ask for the
framework-native one.

```
get_examples({ name: "Dialog", framework: "react" })
```

### 4. Write the code

Compose the snippet into the real feature — here, a destructive confirm
dialog:

```tsx
<Dialog.Root>
  <Dialog.Trigger asChild>
    <Button variant="destructive">Delete account</Button>
  </Dialog.Trigger>
  <Dialog.Content>
    <Dialog.Title>Delete account?</Dialog.Title>
    <Button variant="destructive">Confirm</Button>
  </Dialog.Content>
</Dialog.Root>
```

### 5. `validate_usage` — verify before finishing

Run the deterministic rule engine over the diff. It catches hardcoded colors,
invalid props, raw `@ark-ui`/`@zag-js` imports, bad `data-part` overrides, and
hand-rolled reimplementations of an existing primitive:

```
validate_usage({ code: "…", framework: "react" })
→ { findings: [] }   // green — safe to finish
```

A finding looks like this — fix it and re-run, not once-and-done:

```json
{
  "ruleId": "moderno/no-hardcoded-color",
  "severity": "error",
  "message": "Hardcoded color '#ff0000' bypasses the token contract.",
  "suggestion": "var(--destructive)"
}
```

**Before finishing any task that touches Moderno components, run
`validate_usage` and fix everything it reports.**

## Golden rules

- **Components are never edited.** They're themed via CSS variables and varied
  via props — never fork component markup or write per-component CSS.
- Reference contract slots (`--background`, `--primary`, `--radius`, …) — never
  hardcode hex colors, px spacing, or ms durations.
- Put brand values in a theme, not in `@moderno/tokens`.
- Keep `:root` light / `.dark` dark — don't invent a third theming mechanism.
- `@moderno/css` is the only public CSS specifier — never import internal
  paths.

These are the same rules [`moderno init`](/en/installation) writes into your
project's `AGENTS.md` (and, with `--claude`, a `CLAUDE.md` twin), so an agent
that reads either file lands on this same contract without you repeating
yourself.

## For human contributors

Agents and humans share one rule engine. [`@moderno/lint`](https://www.npmjs.com/package/@moderno/lint)
runs the exact checks behind `validate_usage` as an ESLint plugin and a CLI
(`moderno-lint`), so a PR from a person and a PR from an agent are held to the
same standard — and CI catches what either one misses.
